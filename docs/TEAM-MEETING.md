## 概述

7zi Studio AI 团队支持完整的会议系统，用于：

- ✅ 团队沟通协调
- ✅ 方案讨论决策
- ✅ 进度同步汇报
- ✅ 问题研讨解决
- ✅ 投票表决

### 会议参与方

| 角色 | 权限 | 说明 |
|------|------|------|
| 🧑 主人 | 全部 | 可以参加/旁观/否决 |
| 🤖 主管 | 主持 | 组织和主持会议 |
| 📋 子代理 | 参与 | 讨论和投票 |

## 会议类型

### 1. 每日站会 (Daily Standup)

**频率**: 每天  
**时长**: 15-30 分钟  
**参与者**: 全体子代理

**议程**:
1. 昨日完成工作
2. 今日计划工作
3. 遇到的问题和阻塞

**示例**:
```
主管："开始每日站会"

智能体世界专家:
- 昨日：完成市场趋势分析
- 今日：进行竞争格局研究
- 问题：无

架构师:
- 昨日：完成系统架构设计
- 今日：开始数据库设计
- 问题：需要确认数据量级

...
```

---

### 2. 规划会 (Planning Meeting)

**频率**: 按需  
**时长**: 30-60 分钟  
**参与者**: 相关子代理

**议程**:
1. 任务背景和目标
2. 方案讨论
3. 任务分解
4. 时间估算
5. 责任分配

**示例**:
```
主管："召开规划会，讨论电商网站开发"

1. 目标：2 周内完成电商网站 MVP
2. 方案：
   - 架构师：微服务架构
   - 设计师：移动端优先
3. 任务分解：
   - 前端：商品列表、购物车、下单
   - 后端：用户、订单、支付
4. 时间：2 周
5. 分配：
   - Executor：前端开发
   - 系统管理员：后端部署
```

---

### 3. 问题研讨会 (Problem Solving)

**频率**: 按需  
**时长**: 30-90 分钟  
**参与者**: 相关专家

**议程**:
1. 问题描述
2. 根因分析
3. 方案讨论
4. 决策制定
5. 行动计划

**示例**:
```
主管："召开问题研讨会，讨论性能问题"

1. 问题：首页加载时间>5 秒
2. 根因分析:
   - 测试员：发现图片未压缩
   - 架构师：数据库查询未优化
3. 方案:
   - 图片压缩 + CDN
   - 添加缓存层
   - 数据库索引优化
4. 决策：全部实施
5. 计划:
   - Executor：图片压缩 (今天)
   - 系统管理员：CDN 配置 (明天)
   - 架构师：数据库优化 (后天)
```

---

### 4. 评审会 (Review Meeting)

**频率**: 按需  
**时长**: 30-60 分钟  
**参与者**: 相关子代理

**议程**:
1. 成果展示
2. 质量评审
3. 改进建议
4. 通过/修改决定

**示例**:
```
主管："召开评审会，评审登录功能"

Executor 展示:
- 功能实现完成
- 单元测试通过
- 代码审查通过

评审意见:
- 测试员：需要添加边界测试
- 架构师：密码强度检查需要加强
- 设计师：错误提示可以更友好

决定：修改后重新评审
```

---

### 5. 投票决策会 (Voting)

**频率**: 按需  
**时长**: 15-30 分钟  
**参与者**: 相关子代理或全体

**议程**:
1. 提案说明
2. 方案对比
3. 投票表决
4. 结果宣布

**示例**:
```
主管："投票决定数据库选型"

提案:
A. PostgreSQL - 功能强大，支持复杂查询
B. MongoDB - 灵活，适合快速迭代
C. MySQL - 成熟稳定，生态完善

投票结果:
- PostgreSQL: 5 票
- MongoDB: 3 票
- MySQL: 3 票

决定：选用 PostgreSQL
```

## 会议流程

### 标准流程

```
1. 主管宣布会议开始
         ↓
2. 介绍议程和规则
         ↓
3. 按议程进行讨论
         ↓
4. 记录和总结
         ↓
5. 形成决议或行动计划
         ↓
6. 宣布会议结束
         ↓
7. 发送会议纪要
```

### 详细流程

#### 1. 会议开始

```
主管："各位，现在开始 [会议类型]，主题是 [主题]"
主管："会议预计 [时长]，议程如下："
主管："1. ..."
主管："2. ..."
主管："3. ..."
```

#### 2. 讨论进行

```
主管："首先讨论第一项议程：[议题]"
主管："[子代理名称]，请先分享你的观点"

子代理："我认为..."
子代理："我建议..."

主管："其他人有什么看法？"
```

#### 3. 形成决议

```
主管："综合大家的意见，我们决定..."
主管："同意的请回复 ✅"
主管："反对的请回复 ❌"
主管："决议通过/需要进一步讨论"
```

#### 4. 会议结束

```
主管："今天的会议到此结束"
主管："行动计划如下："
主管："1. [任务] - [负责人] - [截止时间]"
主管："2. ..."
主管："会议纪要将发送到 [渠道]"
```

## 投票决策

### 投票方式

#### 1. 简单多数

```
规则：超过 50% 同意即通过

示例:
主管："同意的请回复 ✅"
主管："反对的请回复 ❌"
主管："弃权请回复 ⭕"

结果:
✅ 7 票
❌ 3 票
⭕ 1 票

决定：提案通过
```

#### 2. 加权投票

```
规则：不同角色权重不同

权重:
- 主人：5 票
- 主管：3 票
- 专家子代理：2 票
- 普通子代理：1 票

示例:
提案：数据库选型
- PostgreSQL: 主人 (5) + 架构师 (2) + 系统管理员 (1) = 8 票
- MongoDB: 咨询师 (2) + Executor (1) = 3 票

决定：PostgreSQL 胜出
```

#### 3. 一票否决

```
规则：主人可以一票否决任何提案

示例:
主管："提案获得 9 票支持，1 票反对"
主人："我否决这个提案"
主管："提案被否决，需要重新讨论"
```

### 投票 API

```typescript
// 发起投票
const vote = await teamMeeting.vote({
  topic: '数据库选型',
  options: ['PostgreSQL', 'MongoDB', 'MySQL'],
  voters: ['architect', 'sysadmin', 'executor'],
  deadline: '2026-03-07 18:00'
});

// 投票
await vote.cast({
  voter: 'architect',
  option: 'PostgreSQL'
});

// 查看结果
const result = await vote.result();
// { PostgreSQL: 5, MongoDB: 3, MySQL: 3 }

// 宣布结果
await vote.announce();
```

## 最佳实践

### 1. 会前准备

```
✅ 好的准备:
- 提前发送议程
- 准备相关材料
- 确认参与者时间
- 设定明确目标

❌ 不好的准备:
- 临时通知
- 没有议程
- 目标不明确
```

### 2. 时间控制

```
✅ 好的时间管理:
- 准时开始
- 按议程进行
- 控制发言时间
- 准时结束

❌ 不好的时间管理:
- 等人到齐
- 跑题讨论
- 超时严重
```

### 3. 有效讨论

```
✅ 好的讨论:
- 聚焦议题
- 基于事实
- 尊重不同意见
- 寻求共识

❌ 不好的讨论:
- 人身攻击
- 固执己见
- 无休止争论
- 没有结论
```

### 4. 跟进执行

```
✅ 好的跟进:
- 明确责任人
- 设定截止时间
- 定期检查进度
- 及时反馈

❌ 不好的跟进:
- 责任不清
- 没有截止
- 无人检查
- 不了了之
```
