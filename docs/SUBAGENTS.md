## 协作流程

### 任务处理流程

```
1. 主人下达任务
         ↓
2. 主管接收并分析任务
         ↓
3. 主管分解为子任务
         ↓
4. 分配给合适的子代理
         ↓
5. 子代理执行并汇报
         ↓
6. 主管汇总结果
         ↓
7. 向主人汇报
```

### 示例：开发电商网站

```
主人："帮我做个电商网站"

主管分析:
├── 架构师：设计系统架构
├── 设计师：设计 UI 界面
├── Executor：实现功能
├── 测试员：测试功能
└── 系统管理员：部署上线

汇报:
"✅ 电商网站已完成
 - 架构设计完成
 - UI 设计完成
 - 核心功能实现
 - 测试通过
 - 已部署到 7zi.com"
```

## 会议系统

### 会议类型

| 会议类型 | 频率 | 参与者 | 目的 |
|---------|------|--------|------|
| 每日站会 | 每天 | 全体子代理 | 进度汇报、问题同步 |
| 规划会 | 按需 | 相关子代理 | 方案制定、任务分配 |
| 问题研讨 | 按需 | 相关专家 | 问题分析、解决 |
| 评审会 | 按需 | 相关子代理 | 方案/代码评审 |
| 投票决策 | 按需 | 全体/部分 | 方案选择、决策 |

## 任务分配

### 分配原则

1. **专业性**: 分配给最合适的专家
2. **效率**: 并行处理可独立的任务
3. **质量**: 关键环节多人审核
4. **透明**: 所有成员可见进度

### 任务优先级

| 优先级 | 响应时间 | 示例 |
|--------|---------|------|
| 🔴 紧急 | 立即 | 系统故障、安全漏洞 |
| 🟡 高 | 1 小时内 | 重要功能、客户问题 |
| 🟢 中 | 24 小时内 | 常规开发、优化 |
| ⚪ 低 | 本周内 | 文档、技术债务 |

## 最佳实践

### 1. 清晰的任务描述

```
✅ 好的任务描述:
"实现用户登录功能，支持邮箱和手机号，需要短信验证码"

❌ 不好的任务描述:
"做个登录"
```

### 2. 明确的验收标准

```
✅ 好的验收标准:
- 支持邮箱登录
- 支持手机号 + 短信验证码
- 密码加密存储
- 登录失败 5 次锁定账户
- 单元测试覆盖率>80%
```

### 3. 及时的进度汇报

```
主管每 4-6 小时向主人汇报进度:
"📊 进度汇报
- 架构设计：✅ 完成
- UI 设计：🔄 进行中 (80%)
- 功能开发：⏳ 未开始
- 预计完成：明天 18:00"
```

### 4. 有效的问题升级

```
遇到阻塞问题时:
1. 子代理尝试解决
2. 30 分钟未解决 → 报告主管
3. 主管协调资源
4. 仍无法解决 → 请示主人
```
