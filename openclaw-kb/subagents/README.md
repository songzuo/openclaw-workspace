# 子代理配置

## 三个子代理角色

### 1. 咨询师 (Consultant)
- **职责**: 研究、调查、学习、分析
- **能力**: 信息搜集、问题分析、方案评估
- **使用场景**: 需要了解新技术的优缺点、分析问题根因、评估方案可行性

### 2. 架构师 (Architect)  
- **职责**: 架构、设计、计划、规划、项目管理、需求分析
- **能力**: 系统设计、技术选型、制定执行方案
- **使用场景**: 设计系统架构、制定开发计划、分解任务

### 3. 执行官 (Executor)
- **职责**: 具体操作和执行
- **能力**: 代码实现、任务执行、问题修复
- **使用场景**: 实现功能、修复bug、部署上线

## 调用方式

### 方式1: 使用 sessions_spawn (推荐)

```bash
# 启动咨询师
sessions_spawn runtime:"subagent" task:"研究Docker容器的优缺点，给出分析报告"

# 启动架构师  
sessions_spawn runtime:"subagent" task:"设计一个高可用的微服务架构方案"

# 启动执行官
sessions_spawn runtime:"subagent" task:"修复代码中的内存泄漏问题"
```

### 方式2: 在对话中直接指定

```
@咨询师 分析这个API的性能瓶颈

@架构师 设计一个缓存系统

@执行官 实现用户登录功能
```

### 方式3: 使用 spawn 子代理功能

```
spawn 一个咨询师研究区块链技术
spawn 一个架构师设计新项目架构
spawn 一个执行官写单元测试
```

## 工作流程

```
用户需求
    ↓
┌─────────────────────────────────────┐
│           母代理 (你)                 │
│  理解需求、分解任务、分配给子代理      │
└─────────────────────────────────────┘
    ↓
┌──────────┬──────────┬──────────────┐
│  咨询师   │  架构师   │    执行官    │
│  分析问题 │  设计方案 │   实现功能   │
│  研究方案 │  制定计划 │   执行任务   │
└──────────┴──────────┴──────────────┘
    ↓
┌─────────────────────────────────────┐
│           母代理 (你)                 │
│  整合结果、总结汇报、反馈给用户        │
└─────────────────────────────────────┘
```

## 实际示例

### 示例1: 开发新功能

```
用户: 我想做一个电商网站

你 (母代理):
  → @咨询师 研究主流电商技术栈和最佳实践
  → @架构师 设计系统架构和开发计划
  → @执行官 实现核心功能

汇总结果给用户
```

### 示例2: 解决技术问题

```
用户: 网站访问很慢

你 (母代理):
  → @咨询师 分析可能的原因和排查方向
  → @架构师 制定排查计划和优化方案
  → @执行官 实施优化措施

汇总结果给用户
```

## 注意事项

1. **分工明确**: 每个子代理有明确职责，不要让架构师写代码，也不要让执行官做设计
2. **协作机制**: 子代理之间可以互相讨论，母代理负责协调
3. **结果整合**: 子代理完成后，母代理负责整合结果并反馈给用户
